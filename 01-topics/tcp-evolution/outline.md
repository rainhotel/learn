# TCP 发展历史和发展历程 Mastery Outline

## Phase 0: Positioning And Scope

- 研究问题：TCP 为什么会不断演进
- 核心主线：从“能连通”发展到“能在复杂网络和 Web 场景中高效稳定运行”
- 学习完成标准：
  - 能按时间线复述关键改进
  - 能把改进与网络问题一一对应
  - 能解释 TCP 在现代 Web 中的作用与边界

## Phase 1: Map The Field

- 建立两条主线：
  - 协议内部主线：可靠传输、流量控制、拥塞控制、丢包恢复、连接建立优化
  - 应用外部主线：HTTP/1.0 到 HTTP/1.1，再到 HTTP/2 对 TCP 的需求变化
- 关键时间节点：
  - 1981-09：RFC 793，TCP 正式标准化
  - 1989-10：RFC 1122，主机实现要求更明确
  - 1992-05：RFC 1323，窗口扩大和时间戳
  - 1996-10：RFC 2018，SACK 选择确认
  - 2009-09：RFC 5681，总结慢开始、拥塞避免、快重传、快恢复
  - 2013-04：RFC 6928，初始拥塞窗口增大
  - 2014-12：RFC 7413，TCP Fast Open
  - 2015-05：RFC 7540，HTTP/2 推动 TCP 使用方式变化
  - 2023-08：RFC 9438，CUBIC 规范化

## Phase 2: Foundations

- TCP 的基础职责：
  - 面向连接
  - 可靠字节流
  - 流量控制
  - 拥塞控制
- 必须区分的概念：
  - 流量控制 vs 拥塞控制
  - 丢包恢复 vs 拥塞避免
  - 持久连接优化的是连接复用，不等于 TCP 本身变成多路复用协议

## Phase 3: Core Mechanisms

- 为什么三次握手重要
- 为什么慢开始是必须的
- 为什么高带宽高时延链路逼出了窗口扩大和时间戳
- 为什么累计确认在多丢包下不够用，从而需要 SACK
- 为什么 Web 小对象请求推动了 Fast Open 和更大的初始拥塞窗口

## Phase 4: Patterns And Trade-Offs

- 演进规律 1：每次改进几乎都源于某种“现实网络不再适配旧默认值”
- 演进规律 2：改进常在吞吐、时延、公平性、复杂度之间折中
- 典型权衡：
  - Keep-Alive 降低建连成本，但可能放大队头阻塞影响
  - HTTP/2 提高单连接利用率，但 TCP 层丢包仍会阻塞该连接上所有流
  - Fast Open 降低首包时延，但引入部署和安全方面的额外考虑
  - 更激进的拥塞控制提升吞吐，但必须兼顾公平性和稳定性

## Phase 5: Deliberate Practice

- 练习 1：解释 TCP 为什么不是“一次设计永久不变”
- 练习 2：解释 HTTP/1.1 Keep-Alive 为什么依赖 TCP 的可靠连接
- 练习 3：解释为什么 HTTP/2 仍然可能受 TCP 队头阻塞影响
- 练习 4：解释分布式高并发 Web 系统为什么依然需要 TCP

## Phase 7: Explain, Teach, Defend

- 能回答：
  - 为什么 TCP 在 Web 中没有被完全替代
  - 为什么 HTTP/2 改了很多，但依旧没绕开 TCP 的约束
  - 为什么现代高并发服务强调 TCP 参数和拥塞控制选择

## Phase 8: Frontier And Taste

- 了解 TCP 的现代边界：
  - CUBIC 成为主流高带宽场景选择之一
  - QUIC 之所以出现，很大程度上是为了绕开 TCP 在应用多路复用上的限制
- 形成判断：
  - TCP 不是“过时”，而是“稳定、成熟、广泛部署，但在低时延多路复用上有天然包袱”
