## Answered Questions

### 为什么 TCP 不是“早就定型、后面没什么变化”的协议？

- 结论：TCP 的基本语义稳定，但性能与恢复机制一直在演进。
- 依据：从 RFC 1323、RFC 2018、RFC 6928、RFC 7413 到 RFC 9438，都在修正不同年代的新问题。
- 例子：高带宽高时延环境逼出了窗口扩大；Web 短流则推动了更大的初始窗口和 Fast Open。

### HTTP/1.1 的 Keep-Alive 是不是 TCP 本身的一次大改版？

- 结论：不是。
- 依据：它更准确地说是应用层对 TCP 连接复用能力的利用。
- 例子：HTTP/1.1 让多个请求尽量复用同一连接，从而减少重复握手成本。

### HTTP/2 既然多路复用了，为什么还会受 TCP 限制？

- 结论：因为 HTTP/2 的多个流通常仍跑在一条 TCP 连接上。
- 依据：应用层流可以并发，但 TCP 层一旦发生丢包，重传与有序交付仍会影响整条连接。
- 例子：一个数据段丢失时，后续字节即使属于别的流，也要等 TCP 恢复。

### 分布式高可用高并发 Web 为什么还离不开 TCP？

- 结论：因为它仍然提供最通用、最成熟、最广泛部署的可靠连接语义。
- 依据：服务间 RPC、数据库连接、反向代理、连接池、负载均衡几乎都依赖 TCP。
- 例子：Nginx 到应用服务器、应用到 MySQL、服务网关到下游服务，底层常见都是 TCP。
