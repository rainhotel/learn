# TCP 发展历史和发展历程 Notes

## Key Concepts

### 1. TCP 标准化

- 定义：TCP 在 1981 年 9 月通过 RFC 793 形成经典基础规范。
- 为什么重要：这标志着互联网可靠传输有了统一的连接、确认、重传与流量控制框架。
- 常见误区：把 TCP 看成“一开始就设计完美”。实际上后续几十年的大量 RFC 都是在修补真实网络中的性能与效率问题。

### 2. 拥塞控制的出现

- 定义：当网络中发送方过多、发送过快时，TCP 需要根据网络状态主动减速。
- 为什么重要：如果没有拥塞控制，网络会出现拥塞崩溃，吞吐反而下降。
- 常见误区：把流量控制当成拥塞控制。前者保护接收方，后者保护网络。

### 3. 慢开始与拥塞避免

- 定义：连接初期先小心试探网络容量，再逐步增加发送窗口。
- 为什么重要：它让 TCP 从“能传”走向“稳定高效地传”。
- 常见误区：误以为“慢开始”就是传得慢。它其实通常是指数增长，只是相对激进直发更保守。

### 4. 高性能扩展

- 定义：RFC 1323 在 1992 年 5 月提出窗口扩大和时间戳等扩展，用于高带宽高时延链路。
- 为什么重要：原始 16 位窗口在长肥网络中很快成为瓶颈。
- 常见误区：以为 TCP 性能差只是实现问题。很多时候是旧协议默认值不适应新链路规模。

### 5. SACK 选择确认

- 定义：RFC 2018 在 1996 年 10 月引入 SACK，让接收方能告诉发送方“哪些块已经收到”。
- 为什么重要：多段丢包时，发送方不必因累计确认的局限而重复重传过多数据。
- 常见误区：以为确认机制从一开始就足够精细。原始累计确认在复杂丢包下效率并不高。

### 6. 面向 Web 的连接复用

- 定义：HTTP/1.1 的持久连接让多个请求复用同一条 TCP 连接，避免每个对象都重新三次握手。
- 为什么重要：它直接降低了 Web 页面加载中的建连成本。
- 常见误区：把 Keep-Alive 说成“TCP 新增字段”。它更准确地说是 Web 对 TCP 连接复用方式的利用。

### 7. 初始拥塞窗口增大

- 定义：RFC 6928 在 2013 年 4 月将 TCP 初始拥塞窗口提高到更合理的范围，改善短流性能。
- 为什么重要：现代 Web 有大量小对象传输，过小的初始窗口会让页面首屏变慢。
- 常见误区：以为这只是参数调整。它本质上反映了 Web 负载结构对 TCP 默认行为的反向塑造。

### 8. TCP Fast Open

- 定义：RFC 7413 在 2014 年 12 月提出在后续连接中缩短建连与首包传输之间的等待。
- 为什么重要：它针对的是 Web 小请求中“三次握手先花一轮 RTT”的时延痛点。
- 常见误区：把它理解成“完全取消三次握手”。实际上它是有条件地减少等待成本。

### 9. HTTP/2 与 TCP 的新矛盾

- 定义：HTTP/2 在 2015 年 5 月支持在一条连接上并发多个流，显著改善应用层队头阻塞。
- 为什么重要：它进一步提升了单连接利用率，减少了浏览器开多连接的需求。
- 常见误区：以为 HTTP/2 已经彻底解决阻塞。实际 TCP 层一旦丢包，连接上的多个流仍会一起受影响。

### 10. 现代拥塞控制与 CUBIC

- 定义：CUBIC 面向高速长距离网络优化窗口增长行为，并在 RFC 9438 于 2023 年 8 月规范化。
- 为什么重要：这说明 TCP 仍在持续演进，而不是停留在教科书里的 Tahoe/Reno。
- 常见误区：把 TCP 拥塞控制理解成固定四步法。现实系统里算法已更丰富。

## Mental Model

- 一张最值得记住的脑图：
  - TCP 的演进不是线性堆功能，而是被三股力量反复推动：
  - 第一股力量是网络规模扩大，逼出拥塞控制和更好的丢包恢复。
  - 第二股力量是链路能力提升，逼出窗口扩大和更现代的拥塞算法。
  - 第三股力量是 Web 业务形态变化，逼出持久连接、短流优化和多路复用协同设计。

## Timeline

### 阶段 1：从可连通到可标准化

- 1981-09，RFC 793 发布，TCP 成为互联网运输层核心协议之一。
- 这一阶段的重点是：建立可靠连接、序号确认、重传和流量控制的基本框架。

### 阶段 2：从可用到不拥塞崩溃

- 1980 年代后期，网络负载增长暴露出拥塞崩溃问题。
- 慢开始、拥塞避免、快重传、快恢复逐步成为 TCP 的核心稳定机制，后在 RFC 5681 中得到系统总结。

### 阶段 3：从普通链路到长肥网络

- 1992-05，RFC 1323 处理高带宽高时延链路上的窗口限制和 RTT 测量问题。
- 这说明 TCP 必须适配“更快但更远”的网络环境。

### 阶段 4：从累计确认到精细恢复

- 1996-10，RFC 2018 引入 SACK。
- 这让 TCP 在多丢包场景中更高效，减少了不必要重传。

### 阶段 5：从通用传输到 Web 时代优化

- HTTP/1.0 短连接暴露出频繁握手成本。
- HTTP/1.1 持久连接缓解了这个问题。
- 2013-04 的 RFC 6928 和 2014-12 的 RFC 7413，分别从初始发送窗口和首包时延角度进一步优化短流体验。

### 阶段 6：从单请求优化到多路复用协同

- 2015-05，HTTP/2 在单 TCP 连接中承载多流。
- 它减少了应用层阻塞，却也把 TCP 层丢包影响集中到了整条连接上。

### 阶段 7：从经典算法到现代调优

- 2023-08，RFC 9438 将 CUBIC 规范化。
- 现代 TCP 更强调在高带宽、长距离和复杂数据中心边缘场景中的吞吐、稳定性和公平性平衡。

## TCP 在分布式高可用高并发 Web 中的作用

### 1. 可靠传输基础

- TCP 为请求和响应提供按序、可靠的字节流传输。
- 对登录、下单、支付、配置同步、服务间 RPC 等业务来说，这种可靠性是底座。

### 2. 连接复用与连接池基础

- 长连接、连接池、反向代理复用连接，本质上都建立在 TCP 连接之上。
- 没有稳定的 TCP 连接，服务端就要频繁重新握手，CPU、时延和端口资源都会受影响。

### 3. 流量整形与系统稳定性

- TCP 的拥塞控制会在网络拥挤时自动收敛发送速率。
- 在多机分布式系统中，这种“自动减速”有助于避免局部网络拥塞迅速演化为整体雪崩。

### 4. 负载均衡与健康连接维护

- LVS、Nginx、Envoy、云负载均衡器都大量处理 TCP 连接建立、复用、关闭和超时。
- 高可用设计不仅是“服务副本多”，也包括“连接管理稳”。

### 5. 局限与边界

- TCP 以单连接可靠有序为优势，但在多路并发、跨流独立恢复上存在天然限制。
- 这也是后续 QUIC 受到关注的重要背景，但不意味着 TCP 在 Web 中失去价值。

## Promoted From Problems Or Research

- 从课程设计中沉淀出的知识点：
  - TCP 的历史不能只写 RFC 名称，必须写“问题 - 修改 - 结果”。
  - Web 对 TCP 的影响主要体现为连接使用方式和默认参数优化。
- 从研究中沉淀出的知识点：
  - TCP 的许多关键改进本质上都是“为了更好适配新网络条件”。
